# 📋 111. 오팔 독립 API 플랫폼 아키텍처 상세 설계서 (Opal API Platform)

**날짜: 2026년 3월 9일**  
**버전: v1.0.0 (Master Blueprint)**  
**설계자: 공동감독 김감독 (AI)**  
**검수 대상: 외부 기술 검증 및 이감독 (사용자)**

---

## 1. 서론 (Introduction)
본 설계서는 구글 오팔(Google Opal)의 공식 API 부재 상황을 극복하고, 링크드랍 V2 전용 **'사설 API 게이트웨이'**를 구축하여 안정적이고 표준화된 영상 생산 인프라를 제공하는 것을 목적으로 한다.

## 2. 아키텍처 원칙 (Core Principles)
1.  **완전 격리 (Total Isolation)**: 노트북LM(기존 시스템)과 오팔 엔진의 코드를 물리적·논리적으로 100% 분리한다.
2.  **인증 무결성 (Auth Integrity)**: 브라우저 세션 의존성을 탈피하고, 서버 측 세션 영속성(Persistence) 기술을 도입한다.
3.  **표준화 (Standardization)**: OpenAPI 3.0 명세를 준수하여 향후 공식 API 출시 시 즉시 교체가 가능한 인터페이스를 구축한다.

---

## 3. 기술 스택 (Technical Stack)
| 계층 | 기술 구성 | 비고 |
| :--- | :--- | :--- |
| **Hosting** | Railway.app / Render.com (PaaS) | Docker Container 기반 독립 환경 |
| **API Framework** | FastAPI (Python 3.11+) | 고성능 비동기 처리 및 OpenAPI 자동 생성 |
| **Session Engine** | Playwright (Headless Mode) | 자동 세션 갱신 및 DOM 조작 엔진 |
| **Task Queue** | Celery + Redis (Optional) | 대량 렌더링 요청 순차 처리(Queue) |
| **Security** | Header-based API Key Auth | `X-LD-OPAL-API-KEY` 발급 및 검증 |

---

## 4. 시스템 구조 및 워크플로우 (System Workflow)

### ① API 게이트웨이 레이어
- 외부(LinkDrop Admin/MCP)로부터 정식 API 요청 수신.
- 발급된 API 키의 유효성 및 권한 검증.
- 요청 데이터(`Rule 104` JSON)의 구문 및 표준 준수 여부 검사.

### ② 세션 브릿지 레이어 (Session Persistence)
- **상시 가동 브라우저 컨텍스트**: 서버 내부에서 헤드리스 브라우저가 구글 세션을 유지.
- **세션 자가 치유(Self-Healing)**: 쿠키 만료 전 자동 새로고침 및 백그라운드 활동을 통해 세션 생명력 연장.
- **장애 알림**: 세션 완전 단절 시 관리자에게 Webhook 알림 송신 및 갱신 인터페이스 노출.

### ③ 엔진 제어 레이어 (Node Injection)
- 수신된 설계도를 오팔 내부의 개별 노드(Text, Asset, Audio) 명령어로 변환.
- Playwright를 통한 실시간 DOM 주입 및 렌더링 트리거 실행.
- 렌더링 진행률(Progress) 및 결과물(MP4 URL) 실시간 추적.

---

## 5. 리스크 관리 전략 (Risk Management)

### [장애 대응: 구글 업데이트]
- **문제**: 구글 오팔의 UI/통신 규격 변경.
- **대응**: 
    - **Selector Abstraction**: 하드코딩된 선택자 대신 AI 기반의 유연한 요소 탐지 로직 도입.
    - **Hot-fix Pipeline**: 규격 변경 감지 시 에이전트(김감독)가 즉각 코드를 수정하여 무중단 서비스 지향.

### [안정성: 사설 API의 한계 극복]
- **Rate Limiting**: 구글 계정 보호를 위한 요청 빈도 자동 조절.
- **Shadow Session**: 여러 구글 계정을 풀(Pool)로 관리하여 병렬 처리 및 리스크 분산 가능성 확보.

## 6. 중장기 확장 플랜 (Future Roadmap)
1.  **SaaS화**: 다수의 사용자에게 독립적인 API 키 발급 및 사용량 과금 체계 구축.
2.  **SDK 제공**: 파이썬/자바스크립트용 전용 라이브러리 배포를 통해 플랫폼 생태계 확장.
3.  **공식 API 전환**: 구글 공식 API 출시 시, 엔드포인트 주소만 변경하여 기존 아키텍처 그대로 유지.
