http://localhost:3000/biz/idea    너의 판단은  어떠한가?   완성이 되면  가치 있는 서비스라 생각하는가?

가치 있는 서비스가 될 가능성이 큽니다. 특히 /biz/idea 는 “영상 자막 → 왜 떴는지(정밀 분석) → 유사 시나리오 기반 초안”으로 **유튜브 제작자가 매일 겪는 병목(리서치/벤치마킹/구성 설계)**을 한 화면에서 줄여주기 때문에, 제대로 다듬히면 돈을 낼 이유가 명확합니다.

지금 구조에서 강점(이미 가치가 보이는 부분)

- SRT 기반 정밀 분석: “감”이 아니라 근거 기반으로 왜 잘 됐는지 를 뽑아줌(컨설턴트/PD 리서치 대체).
- 유사 시나리오 DB 연결: “빈 화면에서 쓰기”가 아니라 검증된 패턴에서 시작 하게 함(초안 생산 속도↑).
- 초안을 씬 테이블로 바로 편집: 촬영/편집 단위로 바로 쓰게 만들어 실제 작업 흐름과 맞음.
완성도를 좌우하는 핵심(여기서 승부가 갈림)

- “초안 품질”의 일관성: 유사 시나리오 매칭이 좋아도, 사용자 입장에서는 내 채널 톤/타깃/포맷에 맞는 변환 이 필요합니다. (이게 없으면 ‘좋은 예시 모음’ 느낌으로 끝날 수 있음)
- 편집 메타 확장: 씬마다 화면/자막/컷 길이/B-roll/CTA 같은 실행 지시가 붙으면 “도구”가 아니라 “제작 시스템”이 됩니다.
- 결과물 연결(Export): 최종적으로 촬영 대본/자막 SRT/편집용 컷 리스트/성우 스크립트 로 뽑아주는 순간 사용자가 시간을 절약했다고 체감합니다.
제 판단

- “완성이 되면 가치 있는가?” → 예.
- 다만 성공 조건은 “분석 화면” 자체가 아니라, 사용자가 오늘 업로드할 영상 하나를 더 빨리/더 잘 만들게 해주는지 입니다. 지금 방향(자막→분석→씬 편집)은 그 목표에 정면으로 맞고 있어서, 남은 건 품질/자동화/내보내기 연결을 얼마나 날카롭게 하느냐입니다.

/biz/idea 는 C:\LinkDropV2\packages\tools 쪽 “스킬(파이썬 도구)”을 실제로 사용합니다. 방식은 “프론트에서 직접 호출”이 아니라, Next.js API Route가 Python 스크립트를 spawn해서 실행 하는 구조입니다.

/biz/idea 에서 packages/tools가 쓰이는 대표 구간

- 정밀 분석(벤치마킹 보고서)
  - /api/biz/idea/benchmark-report → packages/tools/skill-0-youtube-fetcher/benchmark_report.py 실행
  - 이 파이썬 스크립트가 Google Gemini 호출까지 담당합니다.
- 유튜브 자막(SRT) 자동 추출 / 댓글 / 크롤링 도 같은 방식으로 skill-0-youtube-fetcher 의 파이썬 스크립트들을 호출합니다. (예: transcript_srt_fetch.py , comments_fetch.py , keyword_crawl.py )
즉, /biz/idea 는 “packages/tools 스킬 기반 파이썬 파이프라인” + “웹 UI/Next API”가 결합된 형태로 돌아가고 있습니다.

# 앞으로의 개발 방향 및 구현안 (추가)

## 목표(한 문장)
자막(SRT)만 있으면 “벤치마킹 근거 리포트 → 유사 시나리오 기반 씬 편집 → 즉시 촬영/편집 가능한 산출물”까지 한 번에 제공한다.

## 사용자 플로우(권장)
1) 자막 입력(유튜브 자동 추출 또는 SRT/TXT 업로드)  
2) 정밀 분석(벤치마킹 보고서 생성)  
3) 유사 시나리오 검색(초안 시드 확보)  
4) 씬 테이블에서 바로 편집(훅/전개/클로징/CTA까지 조정)  
5) 내보내기(촬영 대본 / 자막 / 편집 가이드 / 성우 스크립트)

## 현재 상태(구현된 것 중심)
- 정밀 분석: `/api/biz/idea/benchmark-report`가 `packages/tools/skill-0-youtube-fetcher/benchmark_report.py`를 실행하고, 파이썬이 Gemini로 보고서 생성
- 유사 시나리오: Supabase `scenarios` 테이블에서 후보를 가져온 뒤 단순 유사도(Jaccard)로 베스트 1개 선택
- 시나리오 편집: 씬 테이블 기반 편집 UI(씬별 텍스트 수정 → 전체 스크립트로 자동 합치기)
- 확장 포인트 UI: 화풍/성우 드롭다운 버튼(현재는 UI만 존재, 추후 로직 예정)

## 부족한 부분과 구현안(우선순위)

### 1) 유사 시나리오 검색 품질(가장 중요)
**문제**
- 현재 유사도는 토큰 기반 Jaccard로, “의도/톤/포맷” 유사도를 잘 반영하지 못함
- 최적 후보가 아니라 “최근 데이터 중 단어가 비슷한 것”을 잡을 가능성 존재

**구현안 A: 임베딩 기반 검색 + 필터**
- Supabase에 `scenarios.embedding`(vector) 컬럼을 두고, 입력 텍스트도 임베딩 생성
- `match_scenarios(query_embedding, template_id, top_k)` RPC로 상위 N개를 가져온 후, UI에서 미리보기 선택 제공
- 최소 필터: 템플릿/길이/장르/타깃 연령대(가능하다면)  

**구현안 B: 하이브리드(키워드+임베딩)**
- 임베딩 top_k 결과에 대해 키워드(훅/CTA/주제 키워드) 점수를 추가 가산해 랭킹 안정화

**추가 제안**
- “유사도 %”는 단일 지표로 오해될 수 있어, 추후에는 `주제 유사/톤 유사/포맷 유사`처럼 분해 지표로 표시하는 것이 UX에 유리

### 2) 씬 구조의 “의미 있는 편집 메타”로 진화
**문제**
- 지금은 씬 텍스트만 편집 → 실제 제작(촬영/편집) 관점의 지시가 부족

**구현안(씬 모델 확장)**
- 씬마다 아래 필드를 추가하고, UI에서 토글로 표시(기본은 숨김):
  - `duration_sec`(권장 길이)
  - `shot`(화면 지시: 정면/오버숄더/핸드샷 등)
  - `broll`(삽입 화면)
  - `subtitle`(자막 문구)
  - `sfx_bgm`(효과음/BGM 키)
  - `cta`(행동 유도 문구)
- 저장 포맷: JSON(씬 배열) + 렌더링용 `script_text`(합본 텍스트)를 함께 유지

### 3) “화풍/성우” 드롭다운의 실제 가치 연결
**문제**
- 현재 버튼 UI만 있고, 결과물과 연결이 없음

**구현안**
- 화풍: 영상 제작 파이프라인(예: Opal/이미지/키프레임)과 연결될 스타일 프리셋(키워드 묶음)을 선택
- 성우: TTS/성우 스크립트 생성 포맷에 반영(말투/속도/호흡/강조 규칙)
- 최소 기능: 선택 값이 “내보내기 산출물”에 메타로 포함되도록(나중에 자동화 연결 시 그대로 사용 가능)

### 4) 산출물(Export) 강화 = 결제 이유
**왜 필요한가**
- 사용자는 “좋은 화면”보다 “바로 쓸 파일”에서 시간을 절약했다고 느낀다

**구현안(1차)**
- 촬영 대본 TXT: 씬번호 + 내레이션
- 자막 SRT: 씬별 타임은 아직 자동화 어려우므로 1차는 “자막 원문(씬별 줄)” 형태로 제공
- 편집 가이드 TXT/CSV: 씬번호/자막/화면/B-roll/CTA
- 성우 스크립트: 호흡/강조 지시가 포함된 포맷

### 5) 저장/버전 관리(사용자 편집 보호)
**문제**
- 씬 편집 결과가 브라우저 새로고침/이동 시 소실될 수 있음

**구현안**
- 로컬 임시 저장(LocalStorage) + 서버 저장(Supabase) 2단
- 저장 단위: `idea_sessions`(사용자/비디오/생성 시각) + `draft_versions`(버전/본문/씬 JSON)
- “되돌리기/버전 비교”는 프리미엄 기능으로 자연스럽게 확장 가능

## 기술 설계 제안(구조)

### 서버(Next API / Python)
- Next API는 “입력 검증/권한/요청 라우팅/타임아웃/로그” 역할로 정리
- Python 스킬은 “외부 API(Gemini/YouTube)/데이터 처리/리포트 생성”에 집중
- 장기적으로는 Python을 독립 서비스로 분리해도 되지만, 초기에는 현재처럼 spawn 구조 유지가 개발 속도에 유리

### 데이터(Supabase)
- `scenarios`: 시나리오 원문 + 메타(템플릿/톤/길이/타깃 등) + (추후) embedding
- `idea_sessions`: 사용자별 작업 세션
- `draft_versions`: 씬 JSON + 합본 텍스트 + 선택 메타(화풍/성우)

## 운영/품질(필수 체크리스트)
- 비용 제어: Gemini 호출 횟수/토큰 제한, 캐시(동일 SRT 재호출 방지)
- 에러 UX: “왜 실패했는지”를 사용자가 이해 가능하게(키 누락/할당량/네트워크/응답 파싱)
- 보안: `SUPABASE_SERVICE_ROLE_KEY`, `GOOGLE_API_KEY`는 서버에서만 사용(클라이언트 노출 금지)
- 관측성: 요청별 `request_id`로 Next API 로그 + Python stderr tail 저장
- 테스트: 최소한 API 라우트의 입력 검증/파싱 실패 케이스는 자동 테스트로 커버
