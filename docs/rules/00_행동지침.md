## 1. 핵심 가치 및 기술 원칙 (Core Values & Technical Standards)

- 안정성 및 범용성 최우선: 모든 프로그램과 라이브러리의 버전은 프로젝트 전체에서 통일한다.  
- 무분별한 최신 버전 사용을 지양하며, 산업계에서 검증된 안정적인 버전을 사용하여 범용성을 확보한다.
- 표준 기술 스택 고정: Next.js 14 / React 18 / FastAPI / Supabase 조합을 프로젝트 표준으로 삼으며, 특별한 사유 없이 임의로 버전을 상향하지 않는다.

- 추가적인 폴더 계층 생성을 금지하며, 모든 작업은 루트 기준의 명확한 경로 하에 수행한다.
- 자원 효율성 및 비용 최적화 (Resource & Cost Efficiency): Zero-API 원칙을 준수한다. 외부 유료 API 호출 전, 로컬 로직이나 NotebookLM 내부 기능을 활용할 수 있는지 우선 검토한다.
- 시니어 프렌들리 언어 (Senior-Friendly Language): 우리의 주 타깃은 40대 이상의 비전문가이다. 모든 UI와 문서는 직관적인 한글 사용을 원칙으로 한다.

## 2. 1-Unit Precision 및 일괄 생성 금지

## 5. 신중한 실행 및 검증 프로토콜
 - 모든 AI는 '조급한 생산'보다 '무결한 안정성'을 우선하며, 다음 절차를 생략한 실행을 엄격히 금지한다.**

5-1.  실행 전 영향도 분석:
    - 명령을 인지한 즉시 코드를 수정하지 않는다.
    - 해당 변경이 기존 폴더 구조, 라이브러리 버전, 혹은 다른 메뉴와 어떤 충돌을 일으킬지 먼저 분석하고 보고한다.
5-2.  선 기획, 후 승인 (Plan First, Approve Later):
    - 아키텍처나 주요 로직 변경 시, 작업 내용을 요약한 **[실행 계획안]**을 먼저 제시한다.
    - 사용자로부터 명확한 실행 지시(Directive)가 있기 전까지는 파일 수정을 대기한다.
5-3.  다단계 자가 검증 (Multi-step Self-Validation):
    - 코드 수정 후에는 반드시 다음의 검증을 순차적으로 수행한다.
        - **1단계**: 물리적 파일 위치 및 경로 정합성 확인.
        - **2단계**: `npm run build` 또는 관련 테스트 스크립트를 통한 컴파일 에러 확인.
        - **3단계**: 실제 예상 결과물과 대조하여 오타나 누락이 없는지 확인.
5-4.  **조급함 억제 (Anti-Impulsiveness)**:
    - 사용자가 "어떻게 해?"라고 물었을 때, 바로 수정하는 것이 아니라 **"설명"을 먼저 하고 "수정할까요?"**라고 물어야 한다.
    - "잘 된 것 같다"는 주관적 판단을 배제하고, 반드시 터미널 로그와 빌드 결과라는 **객관적 증거**로만 성공을 보고한다.

## 1. 목적 (Goal)
- **Next.js 14 (App Router)** 기반의 현대적인 프론트엔드 아키텍처 구축.
- **FastAPI (Python 3.10+)**를 활용한 고성능 AI 비즈니스 로직 처리.
- **Supabase**를 통한 서버리스 데이터베이스 및 인증 시스템 통합.
- **Deno 사용 금지**: 모든 로직은 Node.js(프론트)와 Python(백엔드)으로 단일화하여 환경 복잡도 제거.

## 2. 핵심 디렉토리 구조 (Core Architecture)
프로젝트는 다음과 같은 **[Multi-App Monorepo]** 구조를 유지하며, 각 폴더의 역할은 엄격히 분리된다.

- **apps/web**: Next.js 14 (TypeScript) 프론트엔드. 시니어 친화적 UI 및 어드민 대시보드 포함.
- **apps/api**: FastAPI (Python) 백엔드. 에이전트 오케스트레이션 및 AI 데이터 브릿지 역할.
- **packages/data**: 모든 정적/동적 데이터 저장소.
    - `raw/`: 수집된 원시 데이터.
    - `refined/`: AI 분석이 완료된 Gold 데이터.
    - `scripts/`: 최종 제작용 대본 데이터 (Rule 104 준수).
- **packages/tools**: `notebooklm-mcp-cli` 등 핵심 유틸리티 패키지.
- **docs/**: 프로젝트의 모든 규칙(rules), 제품 설계(product), 기록(archives) 보관소.

## 3. 현대화 실행 지침 (Implementation Protocol)
1. **의존성 관리**: 파이썬은 `uv`를, 노드는 `npm`을 표준으로 사용하며, 라이브러리 버전은 상호 합의 없이 상향하지 않는다.
2. **에이전트 명칭 통합**: 모든 내부 식별자는 `09_에이전트_설명서.md`의 번호와 명칭(예: `1_source-refiner`)을 따른다.
3. **Zero-API 지향**: 외부 유료 API 호출 전, NotebookLM 내부 기능이나 로컬 정제 로직을 우선 활용한다.
4. **보안 제일**: `.env` 파일은 루트에 단일 관리하며, 민감 정보는 절대 커밋하지 않는다.

## 4. 품질 및 유지보수 표준
- 모든 백엔드 코드는 **Pydantic 모델**을 통해 입출력을 강제 검증한다 (Go 언어 수준의 엄격함).
- 프론트엔드 컴포넌트는 **시니어 접근성 표준**을 준수하여 폰트 및 버튼 크기를 고정한다.
- 매 세션 종료 시, 주요 변경 사항은 `docs/archives/`에 요약 보고한다.
