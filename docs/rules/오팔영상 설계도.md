# 오팔영상 설계도 — LinkDrop Opal 키프레임 파이프라인

---

## ★ LinkDrop 서비스 핵심 철학

> **AI 사용을 최대로 줄이면서 최고의 결과물을 만들어 내는 시스템**

- AI는 시나리오 생성(1회)과 HTML 렌더링에만 사용
- 화풍 프롬프트, 씬 조합, 파이프라인 로직은 모두 사전 정의된 값 사용
- 반복 실행 구간에서 AI 재해석 개입 = 0
- 비용과 속도는 AI 호출 횟수에 비례 → 호출 최소화가 곧 경쟁력

## 1. 전체 파이프라인 구조

```
사용자 주제 입력 (1회)
        ↓
  [AI 1회] Gemini — 시나리오 생성 (씬 N개)
        ↓
  씬별 반복 (AI 개입 없음)
  ┌─────────────────────────────────────────────┐
  │  art_styles.json[화풍]["system_instruction"] │  ← 고정 (변하지 않음)
  │  + scene_template.format(scene_text=씬텍스트) │  ← 씬 텍스트만 주입
  │              ↓                               │
  │  AppCatalyst generateWebpageStream API        │
  │    (Google Opal 내부 엔진, gemini-2.5-pro)   │
  │              ↓                               │
  │  HTML 수신 → Playwright 렌더 → PNG           │
  └─────────────────────────────────────────────┘
        ↓
  PNG + TTS(edge-tts) → FFmpeg → clip_N.mp4
        ↓
  FFmpeg concat + BGM → final.mp4 (16:9)
        ↓
  [후처리] 9:16 변환 + 캡션 삽입 → final_vertical.mp4
```

## 2. 핵심 설계 결정: AI 개입 없는 프롬프트 직접 주입

### 배경
- 초기에는 매 씬마다 AI가 화풍 프롬프트를 재해석 → 결과 불일치, 토큰 낭비
- **결정**: 화풍 프롬프트를 `art_styles.json`에 완성형으로 사전 정의
- 씬 텍스트만 `{scene_text}` 변수로 주입 → AI 재해석 경로 완전 차단

### 구조
```python
# AI 개입 없음 — 단순 문자열 포맷팅
system_instruction = art_styles["ghibli-real"]["system_instruction"]  # 고정
user_content = art_styles["ghibli-real"]["scene_template"].format(
    scene_text="황혼의 빛이 물드는 대나무 숲"   # 씬 텍스트만 변수
)
```

### 관련 파일
- `packages/tools/skill-2-video-longform1/opal-manager/art_styles.json`
  - `system_instruction`: userInstruction 필드 — 화풍 DNA + HTML 기술 규칙 (고정)
  - `scene_template`: contents[0].text — `{scene_text}` 자리에 씬만 주입
- `opal_test/run_opal_test.py` — 현재 검증된 테스트 스크립트

## 3. AppCatalyst API (확정, 2026-03-17 검증)

### 엔드포인트
```
POST https://appcatalyst.pa.googleapis.com/v1beta1/generateWebpageStream?alt=sse
```

### 요청 구조
```json
{
  "intent": "",
  "modelName": "gemini-2.5-pro",
  "userInstruction": "[art_styles.json system_instruction — 화풍별 고정]",
  "contents": [{
    "role": "user",
    "parts": [{
      "text": "[scene_template.format(scene_text=씬텍스트)]",
      "partMetadata": {"input_name": "text_1"}
    }]
  }],
  "accessToken": "[Bearer 헤더에서 'Bearer ' 제거한 값]"
}
```

### 인증 — 403 해결 확정
- OAuth2 `cloud-platform` 스코프 토큰으로는 **403 ACCESS_TOKEN_SCOPE_INSUFFICIENT**
- **해결**: Playwright `launch_persistent_context` (Chrome 프로필) 실행
  - Opal 앱 로드 시 `getEmailPreferences`, `getLocation`, `checkAppAccess` 요청에서 Bearer Token 자동 캡처
  - `page.on("request", handler)` 이벤트로 캡처
  - 캡처된 토큰 → `session.json` 갱신 → 58분 유효
- Chrome 프로필 경로: `~/.linkdrop-opal/chrome-profile`

## 5. 영상 제작 2단계 로직 (확정)

```
1차 영상: 16:9 (1920×1080)
  - 더빙(edge-tts) 포함
  - 자막 없음
  - 목적: 재사용 가능한 원본 소재 제공
        ↓
2차 영상: 9:16
  - 1차 영상을 재료로 변환
  - 자막 포함
  - 목적: 실제 서비스 최종 완성본 (숏폼)
```

사용자에게 16:9(재사용 소재) + **9:16(최종 완성본)** 모두 제공

## 7. 서비스 아키텍처 — 중장기 목표 (사용자 분산)

### 현재 (중앙집중)
```
LinkDrop 서버 토큰 1개 (1시간 TTL)
  → 모든 사용자가 A1 앱 1개 공유
  → 병목 + 만료 위기 반복
```

### 목표 (사용자 분산)
```
사용자 Google OAuth 로그인 (AppCatalyst 스코프 포함)
  ↓
LinkDrop → 사용자 Drive에 BGL 파일 30개 업로드
  → A1~A30 Opal 앱 자동 생성
  ↓
Supabase users 테이블에 사용자별 app_id 30개 저장
  ↓
이후 모든 영상 생성 → 사용자 본인 토큰 + 본인 앱 사용
  → 안정적 + 무한 확장 + 비용 사용자 부담
```

BGL 구조 (A1 앱 워크플로우 정의):
```json
{
  "nodes": [
    {"type": "embed://a2/generate.bgl.json", "configuration": {"generation-mode": "image-pro"}},
    {"type": "embed://a2/a2.bgl.json#render-outputs"}
  ],
  "edges": [{"from": "generate", "to": "render-outputs"}]
}
```

구현 필요:
1. Google OAuth 동의 화면에 AppCatalyst 스코프 추가
2. BGL 자동 배포 스크립트 (사용자 Drive에 앱 생성)
3. Supabase users 테이블에 per-user app_id 저장
4. refresh_token을 사용자 계정에 귀속

## 8. 동의 모달 (법적 대비, 완성)

목적: "내 허락 없이 Google 계정에 Opal 앱을 만들었다" 민원 대비

| 파일 | 역할 |
|------|------|
| `apps/web/src/data/longform_disclosure.json` | 모달 내용 전체 — 이 파일만 편집 |
| `apps/web/src/app/api/longform/agree/route.ts` | 동의 기록 Supabase 저장 API |
| Supabase `longform_agreements` 테이블 | session_key, IP, UA, 버전, 일시 영구 보관 |

- `_version` 올리면 기존 동의자도 재동의 모달 표시
- 체크박스: "내 Google 계정에 Opal 앱을 생성·운영하는 것에 명시적으로 동의합니다"


---

  현재 구조의 한계점

  씬 텍스트 → 키워드 단순 매칭 → 조합 → API → PNG

  단어 하나만 보고 판단 → 씬의 감정, 계절, 시간대, 긴장감을 못 읽음.

  ---
  고도화 우선순위

  🔴 Tier 1 — 품질 직결 (즉시 효과)

  ① 씬 감정/톤 분류 레이어 추가
  현재: "혈압을 낮추는 시니어 식품" → 키워드 없음 → 기본값 soft_window
  개선: 건강/안정/따뜻 → tone: "calm_warm" → 조명/배경 최적 조합 자동 결정
  씬 텍스트를 감정 카테고리(calm/dramatic/dark/energetic/mystical) 로 먼저 분류 → 더 정확한 파라미터 선택

  ② 검증된 최적 조합 테이블 (Curated Preset)
  "ghibli-real + calm_warm": { "lighting": "golden_hour", "bg": "nature_forest", "cam": "ws" },
  "neo-noir + dramatic":     { "lighting": "shadow_line",  "bg": "urban_cyber",   "cam": "low" }
  직접 테스트해서 좋은 조합을 JSON에 고정 → 일관된 고품질 보장

  ③ negative 프롬프트 활용
  현재: negative 완전 미사용
  개선: style.negatives + n_text (글자 제거) 를 HTML 규칙에 명시적으로 추가

  ---
  🟡 Tier 2 — 안정성/속도 (중기)

  ④ 복수 생성 후 자동 선택 (카메라/조명/배경 3종 변형)
  씬 1개 → user_content만 카메라 파라미터(ws/cu/low)를 다르게 → API 3회 → PNG 3장 → 최고 선택

  핵심 원칙:
  - system_instruction(화풍 DNA)은 고정, user_content만 교체
  - "동일 프롬프트 3번" 아님 → "카메라/구도 3가지 변형"으로 진짜 다른 이미지 확보
  - 오팔은 유료 API가 아님 → API 비용 없음, 시간(약 90초)만 추가 소요

  구현 방식:
  variants = ["ws", "cu", "low"]  # wide / close-up / low angle
  for cam in variants:
      system_instruction, user_content = build_opal_prompt(scene_text, style_id, camera_override=cam)
      # → API 호출 → PNG 저장
  # → 파일 크기 기준 최고 PNG 선택

  선택 기준: 파일 크기 최대 (시각 요소 밀도) + 흰색 지배 픽셀 비율 최소 (빈 화면 제거)
  병렬 실행(⑤)과 함께 구현 시 시간 부담 흡수 가능

  ⑤ 병렬 생성 (씬 20개 → 동시 처리)
  # 현재: 순차 20번 × 30초 = 10분
  # 개선: asyncio.Semaphore(N)으로 N개씩 병렬 처리

  최적 N값은 실제 테스트로 결정:
  - AppCatalyst 계정당 동시 요청 한도 → 미문서화, 실측 필요
  - 429 Too Many Requests 발생 시점이 실질적 한도
  - Playwright PNG 변환은 메모리 고려 → 별도 세마포어 적용 권장
  - 테스트 순서: 전체 동시 → 429 확인 → 한도 이하로 N 조정

  ⑥ HTML 품질 자동 검증 + 재시도
  # PNG 생성 후 체크
  if png_size < 100KB:       # 너무 작으면 빈 화면 의심
      retry()
  if dominant_color == white: # 흰 배경만 나오면 재생성
      retry()

  ---
  🟢 Tier 3 — 완성도 (장기)

  ⑦ 씬 간 색상 일관성 관리
  씬 1: 황금빛 팔레트 → 씬 2: 갑자기 사이버펑크 → 시청자 혼란
  개선: 영상 전체 "마스터 팔레트" 설정 → 모든 씬이 같은 색조 계열 유지

  ⑧ 애니메이션 최적 프레임 포착
  현재: 2초 후 스크린샷 (임의)
  개선: 0.5초 간격으로 5장 찍어서 가장 풍부한 프레임 선택
  HTML 애니메이션은 시간에 따라 품질이 다름

  ⑨ 씬 텍스트 전처리 — 비주얼 키워드 추출
  원문: "혈압을 낮추는 시니어 건강 식품 TOP5"
  전처리: "건강, 식품, 자연, 따뜻함, 신선함" → 이걸로 매칭
  시나리오 텍스트는 정보 전달 목적 → 비주얼 요소가 숨어있음

  ---
  추천 진행 순서

  ① 감정 분류 + 검증 조합 테이블   ← 품질 즉시 향상, 코드 변경 최소
  ② negative 프롬프트 추가          ← 10분 작업, 글자/오류 방지
  ③ 병렬 생성                       ← 씬 20개 테스트 시 필수
  ④ HTML 품질 검증 + 재시도         ← 안정성 확보
  ⑤ 색상 일관성                     ← 영상 완성도의 핵심

    # 시도해볼 수 있는 범위
  asyncio.gather() 전체  → 429 뜨면 제한 확인됨                                                                                                                        
  asyncio.Semaphore(10)  → 10개씩                                                                                                                                      
  asyncio.Semaphore(5)   → 5개씩 보수적                                                                                                                       
  asyncio.Semaphore(3)   → 3개씩 안전     

  art_styles.json system_instruction이 너무 고수준이라 Gemini가 최소한의 SVG만 생성하는 게 문제입니다.   

  현재 오팔 로직   제너레이터(나노 바나나 프로) - 캔바스(아웃풋)  이 구조를
  제너레이터A1-A(프롬프트 분석) - 제너레이터A1-B(나노 바나나 프로) - 캔바스(아웃풋) 이렇게 수정하면  어떻겠는가?

  Opal 화면에서 보이는 것          실제 내부 동작

  Canvas (아웃풋)              =  Gemini가 생성한 HTML 렌더링                                                                                                          
  우리가 Python에서 호출하는 AppCatalyst API URL:                                                                                               https://appcatalyst.pa.googleapis.com/v1beta1/generateWebpageStream                                                                                                  
  요청 바디 안에:                                                                                                                                                      
  "modelName": "gemini-2.5-pro"                                                                                                                                        
                                                                                                                                                                       
  ---                                                                                                                                                                  
  그래서 제안 구조를 다시 번역하면:                                                                                                                                    
 A1-A (프롬프트 분석)    = Gemini Flash  → 씬을 비주얼 스펙으로 변환                                                                                                  
           ↓                                                                                       A1-B (나노 바나나 프로) = Gemini Pro    → 비주얼 스펙을 HTML로 변환
