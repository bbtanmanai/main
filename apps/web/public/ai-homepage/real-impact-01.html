01
loading

Defer what the user can't see yet

By default the browser fetches every image immediately — even ones three screens below the fold. loading="lazy" defers the request until the user scrolls near it. If they never scroll there, the request never happens.

Always include explicit width and height on lazy images. Without them the layout shifts when the image arrives, hurting your CLS score.

Image loading comparison DevTools → Network to verify
Lazy-loaded iframe loading="lazy"

loading="lazy" also works on <iframe> — the browser won't fetch it until the user scrolls near.

HTML 
<!-- Above the fold — load right away -->
<img
  src="hero.jpg"
  alt="Hero"
  loading="eager"
  fetchpriority="high"
  width="1200"
  height="600">

<!-- Below the fold — defer until user scrolls near -->
<img
  src="section-photo.jpg"
  alt="Article photo"
  loading="lazy"
  width="800"
  height="500">

<!-- Works on iframes too -->
<iframe src="map.html" title="Map" loading="lazy"></iframe>
⚠

Never use loading="lazy" on images visible when the page first opens. Deferring them increases your Largest Contentful Paint time, hurting both UX and search rankings.

02
fetchpriority

Tell the browser what matters first

Images are low priority by default — even your hero. fetchpriority="high" moves it to the front of the loading queue from the very start, instead of waiting behind fonts and scripts.

HTML — <head> 
<!-- Preload hero with high priority -->
<link rel="preload" href="hero.jpg"
  as="image" fetchpriority="high">

<!-- Secondary font — low priority, doesn't block rendering -->
<link rel="preload" href="secondary.woff2"
  as="font" fetchpriority="low" crossorigin>

<!-- LCP image: eager prevents deferral, high wins the queue -->
<img
  src="hero.jpg"
  loading="eager"
  fetchpriority="high"
  width="1200" height="600">

<!-- Analytics: deprioritise entirely -->
<script src="analytics.js"
  fetchpriority="low" defer></script>
ℹ

loading="eager" prevents deferral. fetchpriority="high" wins the bandwidth race. They do different things — both belong on your hero image. Fully supported across all major browsers since October 2024.

03
autocomplete

Tell the browser what each field holds

Beyond on and off, the spec defines over 50 token values. When the browser knows what a field expects, it offers the right saved value instantly — or generates one, in the case of new-password.

Personal details & login
Shipping address
Payment cc-name · cc-number · cc-exp · cc-csc
One-time verification code one-time-code

On iOS, autocomplete="one-time-code" + inputmode="numeric" triggers automatic SMS code detection. The entire copy-paste flow becomes a single tap.

HTML — notable tokens 
<!-- current-password: offers saved credentials -->
<input type="password" autocomplete="current-password">

<!-- new-password: offers to generate a strong password -->
<input type="password" autocomplete="new-password">

<!-- OTP: triggers SMS autofill on iOS -->
<input
  type="text"
  autocomplete="one-time-code"
  inputmode="numeric"
  maxlength="6"
  pattern="\d{6}">

<!-- Credit card: three attributes, three jobs -->
<input
  type="text"
  autocomplete="cc-number"
  inputmode="numeric"
  pattern="[0-9\s]{13,19}"
  placeholder="1234 5678 9012 3456">
04
inputmode

Show the right keyboard on mobile

inputmode controls only which virtual keyboard appears — not validation or field behaviour. More flexible than changing type, which affects both. numeric and decimal look the same on desktop but are different keyboards on iOS and Android.

All inputmode values — tap each on mobile
numeric
decimal
tel
email

search
HTML 
<!-- Digits 0–9 only. PINs, postal codes, verification codes -->
<input type="text" inputmode="numeric">

<!-- Digits + decimal separator. Prices, measurements -->
<input type="text" inputmode="decimal">

<!-- OTP field: autocomplete + inputmode working together -->
<input
  type="text"
  autocomplete="one-time-code"
  inputmode="numeric"
  maxlength="6"
  pattern="\d{6}">
ℹ

On a desktop browser all six inputmode values look like a regular text field. Open this page on iOS or Android and tap each field to see the different keyboards.
