## 3. Lock Protocol — 잠금된 결정 변경 절차

아래 항목 중 하나에 **직접** 해당할 때만 발동한다.  
무관한 단순 수정(텍스트, 레이아웃, 버그 수정, 새 컴포넌트 등)은 확인 없이 바로 진행한다.

### 발동 트리거 (9가지)

| 트리거 | 관련 LD |
|--------|---------|
| ThemeProvider defaultTheme / enableSystem 변경 | LD-001 |
| V3 경로 import 추가 | LD-003 |
| LdStatusStepper 색상 변경 | LD-004 |
| 랜딩 콘텐츠를 Supabase/API에서 조회 | LD-005 |
| liquid-glass를 JS 분기로 처리 | LD-006 |
| 한글에 Anton 폰트 적용 | LD-007 |
| checkout에 data-theme 강제 | LD-008 |
| 이메일+비밀번호 인증 복구 | LD-009 |
| users.role 명칭·체계 변경 | LD-011 |

### 발동 시 절차

1. 관련 `rules/LOCKED_DECISIONS.md` 해당 LD 내용을 사용자에게 보여준다
2. `"⚠️ LD-XXX 번복하시려면 'LD-XXX 번복 확인'이라고 말씀해주세요."` 출력
3. 사용자 명시적 허가 후에만 진행

## 4. 서브에이전트 사용 기준

서브에이전트(Agent 도구)는 아래 조건에서만 사용한다:

- 독립적으로 병렬 처리 가능한 작업
- 메인 컨텍스트와 격리가 필요한 작업
- grep/glob으로 충분하지 않은 3회 이상의 넓은 범위 탐색

**서브에이전트 사용 금지:**
- 단순 작업
- 단일 파일 수정
- grep/glob으로 충분한 검색

## 5. BACKLOG 항목 추가 규칙 — 수용 기준 필수

BACKLOG에 새 항목을 추가할 때 **수용 기준(AC, Acceptance Criteria)을 반드시 함께 작성**한다.
AC 없는 항목은 구현을 시작하지 않는다 → 먼저 AC를 작성하고 사용자 확인 후 구현.

```
✅ 올바른 항목
| FEAT-01 | 로그인 버튼 | auth/page.tsx | [ ] 클릭 시 Google OAuth 실행 [ ] 모바일 375px 터치 영역 48dp 이상 |

❌ AC 없는 항목 (구현 시작 불가)
| FEAT-01 | 로그인 버튼 | auth/page.tsx | — |
```

**AC 작성 기준:**
- 체크 가능한 구체적 조건 ("잘 동작함" ❌ / "클릭 시 /checkout으로 이동" ✅)
- 최소 2개 이상
- UI 항목은 모바일(375px) 조건 포함

## 6. 버그 수정 프로토콜 (TDD)

버그 수정 시 반드시 아래 순서를 따른다. 코드 수정을 먼저 하지 않는다.

1. **실패 테스트 작성** — 버그를 재현하는 테스트를 먼저 작성한다. 이 테스트는 반드시 실패해야 한다.
2. **버그 존재 확인** — 테스트 실패로 문제가 실제로 존재함을 검증한다. 테스트가 통과하면 버그가 없거나 테스트가 잘못된 것이다.
3. **코드 수정** — 테스트를 통과시키는 최소한의 수정만 한다.
4. **수정 검증** — 작성한 테스트가 통과하는지 확인한다.
5. **회귀 확인** — 기존 테스트 전체를 실행해 다른 기능이 깨지지 않았는지 확인한다.
6. **반복** — 모든 테스트가 통과할 때까지 3~5를 반복한다. **최대 3회 제한.**

> **핵심 원칙**: 과제를 검증 가능한 목표로 전환한다.
> "버그를 고친다" → "이 테스트를 통과시킨다"

### 3회 초과 시 — 설계자 에스컬레이션 프로토콜

3회 반복 후에도 테스트가 통과하지 않으면:

1. **수정 즉시 중단** — 더 이상 코드를 건드리지 않는다
2. **현재 상태 보고** — 사용자에게 아래 형식으로 보고한다
   ```
   ⚠️ 3회 수정 시도 실패
   문제: [실패한 테스트 + 오류 메시지]
   시도한 수정: [1차·2차·3차 요약]
   판단: 구현 레벨에서 해결 불가 → 설계 재검토 필요
   ```
3. **architect 에이전트 위임** — 설계자가 원인 분석 + 재설계 후 executor에 재위임
4. **재설계 없이 재시도 금지** — architect 승인 없이 4번째 수정 시도 불가

## 7. 구현 완료 프로토콜 — executor 필수 검증 단계

executor가 구현 완료를 선언하기 전 아래 순서를 반드시 밟는다.
**단계를 건너뛰면 BACKLOG 완료 처리 불가.**

```
1. /verify        → TypeScript 타입 에러 0개 확인
2. /qa-feature    → 수용 기준·LD 준수·UI 동작·회귀 위험 확인
3. 둘 다 PASS     → BACKLOG 해당 항목 완료 처리
4. 하나라도 FAIL  → 수정 후 1번부터 재실행
```

신규 UI 컴포넌트 포함 시: `browse` 스킬로 실제 브라우저 렌더링 추가 확인.

## 9. 스킬 체인 워크플로우 — 작업 유형별 고정 순서

단일 스킬 호출이 아니라 **작업 유형마다 정해진 스킬 순서**를 순차 실행한다.  
각 단계를 건너뛰면 다음 단계 진행 불가.

### A. UI 신규 제작 체인

```
1. ui-ux-pro-max   → 설계 방향·컴포넌트 구조 확정
2. 구현             → 코드 작성 (executor)
3. browse          → 실제 브라우저 렌더링·반응형·터치 검증
4. design-audit    → LD 준수·Left Border·콤보색·glass 패턴 검토
```

트리거: `landing*.tsx` 신규 생성 / "새 컴포넌트 만들어줘" / "섹션 추가해줘"

### B. 버그 수정 체인

```
1. investigate     → 원인 파악 (코드 읽기·로그 분석)
2. 수정             → 최소 범위 코드 수정 (TDD 섹션 6 병행)
3. qa              → 수용 기준 재확인·회귀 점검
4. simplify        → 수정 과정에서 생긴 중복·과설계 제거
```

트리거: "에러", "왜 안 되나", "오류", "버그"

### C. 출시 전 체인

```
1. security-review → XSS·인증 우회·SQL injection 점검
2. review          → 코드 품질·컨벤션 최종 점검
3. ship            → 배포·PR 생성
```

트리거: "배포해줘", "푸시해줘", "PR 만들어줘"

### D. 랜딩 페이지 완성 체인

```
1. bencium-innovative-ux-designer → 크리에이티브 UX 방향
2. 구현                            → 코드 작성
3. browse                          → 실제 렌더 확인 (모바일 375px 필수)
4. design-audit                    → 디자인 원칙 준수 검토
5. landing-report                  → 전환율·UX 이슈 분석
```

트리거: `landing1*.tsx` 수정 완료 / "랜딩 완성했어"

### E. 코드 정리 체인

```
1. simplify        → 중복·과설계 제거
2. review          → 최종 품질 점검
3. bencium-code-conventions → Next.js·TypeScript·Supabase 컨벤션 확인
```

트리거: "코드 정리해줘", "리팩터링", "깔끔하게 해줘"

## 10. 고가치 스킬 — 필수 활용 기준

자주 누락되지만 품질에 직접 영향을 주는 스킬. 아래 조건 충족 시 **반드시** 호출한다.

| 스킬 | 필수 호출 조건 | 이유 |
|------|----------------|------|
| `browse` | UI 파일 수정 완료 시 항상 | TypeScript PASS ≠ UI 정상. 실제 렌더·반응형·터치 검증 필수 |
| `simplify` | 구현 완료 직후 항상 | "시니어가 보면 왜 복잡하지?" 방지. 중복·과추상화 제거 |
| `security-review` | ship 전 항상 | XSS·OAuth 우회·API 노출 사전 차단 |
| `landing-report` | 랜딩 배포 후 | 전환율·UX 병목 데이터 기반 파악 |
| `health` | 주 1회 또는 대형 리팩터 후 | 코드베이스 전반 상태 점검 (타입 오류·미사용 의존성 등) |
