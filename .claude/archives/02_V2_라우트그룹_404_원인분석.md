# 02 — Route Group 구조 오류로 인한 /member/dashboard 404 분석

> 발생일: 2026-04-25
> 증상: 구글 OAuth 로그인 후 `/member/dashboard` → Next.js 404 페이지

---

## 증상

- `curl http://localhost:3000/member/dashboard` → 307 (proxy 정상 동작)
- 브라우저에서 로그인 후 `/member/dashboard` → `404 | This page could not be found.`
- `.next/server/app/(member)/dashboard/` 디렉토리는 존재
- `middleware.ts` 없음 (Next.js 16 `proxy.ts` 사용)

처음에는 auth 콜백 쿠키 설정 문제, proxy.ts 이름 문제 등으로 오인해 여러 번 헛방향으로 수정.

---

## 근본 원인

### Next.js Route Group의 URL 비가시성

Next.js App Router에서 괄호 디렉토리 `(groupName)`은 **레이아웃 논리 분리용**이며
**URL에 아무것도 추가하지 않는다.**

```
app/(member)/dashboard/page.tsx  →  URL: /dashboard        ← 실제 생성된 라우트
app/member/dashboard/page.tsx    →  URL: /member/dashboard ← 의도한 라우트
```

### 실제로 생성된 URL (수정 전)

| 파일 경로 | 생성된 URL | 의도한 URL |
|----------|-----------|-----------|
| `app/(member)/dashboard/page.tsx` | `/dashboard` | `/member/dashboard` |
| `app/(member)/content/page.tsx` | `/content` | `/member/content` |
| `app/(member)/referral/page.tsx` | `/referral` | `/member/referral` |
| `app/(partner)/earnings/page.tsx` | `/earnings` | `/partner/earnings` |
| `app/(partner)/network/page.tsx` | `/network` | `/partner/network` |
| `app/(instructor)/courses/page.tsx` | `/courses` | `/instructor/courses` |
| `app/(instructor)/students/page.tsx` | `/students` | `/instructor/students` |
| `app/(checkout)/order/page.tsx` | `/order` | `/checkout/order` |
| `app/(checkout)/payment/page.tsx` | `/payment` | `/checkout/payment` |
| `app/(checkout)/complete/page.tsx` | `/complete` | `/checkout/complete` |

### 왜 404인가

1. proxy.ts가 `/member/dashboard` 요청을 받아 `getUser()` 확인
2. 세션 쿠키가 없으면 `/?auth=1&next=/member/dashboard`로 리다이렉트
3. 로그인 완료 후 `/auth/callback`이 `/member/dashboard`로 리다이렉트
4. Next.js 라우터가 `/member/dashboard`를 탐색 → **라우트 없음 → 404**

실제로 `/dashboard`에 접속하면 대시보드가 렌더되었을 것 (확인 안 함).

### 디버깅이 어려웠던 이유

`.next/server/app/(member)/dashboard/` 디렉토리가 존재하므로
"라우트가 컴파일되어 있다"고 착각하기 쉬움.
Next.js는 소스 디렉토리 구조를 `.next`에 그대로 반영하지만,
**실제 URL 라우팅은 route group을 무시하고 처리**한다.

---

## 수정 방법

Route group을 제거하고 URL 세그먼트를 명시적으로 포함하는 평면 디렉토리로 재구성.

```
수정 전:  app/(member)/dashboard/page.tsx  →  /dashboard
수정 후:  app/member/dashboard/page.tsx    →  /member/dashboard
```

### 변경된 디렉토리 구조

```
(member)/  →  member/
(partner)/  →  partner/
(instructor)/  →  instructor/
(checkout)/  →  checkout/
```

각 그룹의 `layout.tsx`와 모든 `page.tsx`를 함께 이동.
기존 레이아웃 내부의 `href` 값들(`/member/dashboard` 등)은 이미 올바른 형태였으므로
코드 수정 없이 디렉토리 이동만으로 해결.

`.next` 캐시 삭제 후 서버 재시작으로 정상 동작 확인.

---

## 영향받지 않은 Route Group

| 그룹 | 이유 |
|------|------|
| `(public)/` | `/`, `/about`, `/landing/[slug]` — URL에 prefix 불필요, 의도적 구조 |
| `(auth)/` | `/login`, `/signup` — redirect stub이므로 URL prefix 불필요 |
| `admin/(panel)/` | `admin` 디렉토리가 URL prefix 역할 수행, `(panel)`은 내부 레이아웃 분리용 |

---

## 교훈

1. **Route Group은 레이아웃 분리 목적**이다. URL prefix가 필요하면 반드시 실제 디렉토리 세그먼트를 추가해야 한다.
2. **`.next` 컴파일 디렉토리에 라우트가 있다고 URL이 존재하는 게 아니다.** `routes-manifest.json` 또는 `next info`로 실제 URL을 확인해야 한다.
3. Route group 사용 시 URL 경로를 반드시 명시적으로 검증할 것 — `app/(member)/member/dashboard/` 또는 `app/member/dashboard/` 중 선택.
